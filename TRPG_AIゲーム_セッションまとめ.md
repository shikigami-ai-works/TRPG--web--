# TRPG AIゲーム セッションまとめ

## 目的

TRPGをテーマにしたWebアプリケーションを作る前段階として、ゲーム企画のプロットを固めている。
このファイルは、次のセッションや別のAIに現在までの文脈を引き継ぐためのまとめである。

## 元プロット

参照元:

- `D:\TRPGゲームアプリプロット.md`

元プロットの要点:

- TRPGをテーマにしたWebゲーム。
- シナリオは事前に用意する。
- プレイヤーは自由にキャラクターを作成し、シナリオを遊ぶ。
- シナリオごとに必要なNPC数をあらかじめ決める。
- NPCはAIで生成する。
- NPCは性別、年齢、役割、性格を持つ。
- プレイヤー以外の探索者もAIで生成する。
- NPCの行動は、現在の場面とNPCの性格からAIに判断させる構想だった。
- シナリオをクリアしたキャラクターは次のシナリオに引き継げる。
- シナリオ途中でキャラクターがロストした場合、基本的に復元できない。
- 課金によってロストキャラを復元する案があった。
- 好感度システムを持つNPCがいる。
- 好感度が一定以上のNPCは、シナリオクリア時に「連絡先」を入手できる。
- 連絡先を持つNPCは、次のシナリオに1人同伴できる。
- 同伴NPCもロストする。
- プレイヤーキャラとNPCはシナリオ内で成長する。
- シナリオ終了時、出来栄えに応じてボーナスポイントを得る。
- ボーナスポイントでプレイヤーキャラを強化できる。
- 死に関わる判定は本来の確率より甘めにする。
- アイテムはシナリオ内で入手でき、クリア時に持っていれば持ち帰れる。
- 所持アイテムは次のシナリオに持ち込める。
- アイテムを連絡先NPCに持たせて同伴させることも可能。

## 会話で決まった大きな方針

### AIの役割

AIはゲームマスター型にはしない。
AIは主にロール側の補助役として使う。

つまり、AIに任せるのは以下のような領域:

- NPCの台詞生成
- NPCの感情表現
- NPCの性格に沿った反応
- NPC行動候補の理由づけ
- 後日談エピソードの文章生成

ゲーム側が管理すべき領域:

- シナリオ進行
- クリア条件
- ロスト条件
- ダイス結果
- 報酬
- 隠し情報の開示条件
- ゲームバランス

この方針により、AIの自由度を活かしつつ、ゲームとしての整合性を保つ。

### ゲームの核

現時点でのゲームの核:

> プレイヤーは探索者を作り、AIが演じるNPCと関係を築きながらシナリオを攻略する。
> 出会い、成長、ロスト、記録、連絡先、再同行によって、物語が次の冒険へ持ち越される。

この企画は単なるシナリオ攻略ゲームではなく、キャラクターとNPCとの関係を積み重ねるゲームとして見る方が強い。

## 収益設計で決まったこと

ロスト復活課金を中心にするより、遊び方や保存領域を拡張する課金の方がよい、という方針になった。

現在の収益候補:

- プレイヤーキャラクター枠の追加
- NPC連絡先保存枠の追加
- 有料シナリオ
- キャラクター外見変更アイテム
- キャラクター能力値の振り直しアイテム
- NPCとの後日談エピソード

プレイヤーキャラクター枠:

- 初期枠は3から4枠。
- 追加枠は買い切り。
- ロストしたキャラクターは記録や追憶として残せるようにする。

NPC連絡先枠:

- 連絡先の保存枠を設ける。
- 保存枠の追加を収益構造の一つにする。

シナリオ:

- 無料で追加されるシナリオと、有料で追加するシナリオの二種類を用意する。

能力値振り直し:

- 現在保有している能力値の合計値は変えない。
- 合計値を維持したまま、能力値の配分を振り直す。
- 強さを直接売るのではなく、ビルド変更を売る形にする。

## NPC関連で決まったこと

NPCは好感度を持つ場合がある。

- 好感度が一定以上になると、NPCとの後日談エピソードが開放される。
- 好感度が一定以上の状態でシナリオをクリアすると、そのNPCの連絡先を入手できる。
- 連絡先を持つNPCは、次のシナリオに1人同伴できる。
- 同伴したNPCもロストする。
- ロストしていない限り、NPCも成長する。
- NPCの成長傾向は、そのNPCに割り振られた性格に適したものにする。

## エンディングツリー

クリアしたシナリオは、エンディングがツリー形式で見えるようにする。

- 到達済みのエンディングははっきり見える。
- 未到達のエンディングはぼんやり見える、または詳細が隠される。
- 完全に見えないよりも、輪郭だけ見える方がリプレイ動機を作りやすい。

## 仕様化について

ユーザーは、プロットを固めた後に、それをプログラミングするAIが読みやすい形に変換してから実装に入りたいと考えている。

仕様化とは、ふわっとしたアイデアを以下に分けること:

- いつ起きるのか
- 何が条件か
- 成功したら何が起きるか
- 失敗したらどうなるか
- 例外はあるか

例:

```yaml
npc_contact_unlock:
  trigger: "scenario_clear"
  condition:
    npc.has_affection_system: true
    npc.affection >= npc.contact_unlock_threshold
    npc.is_lost: false
  result:
    add_to_player_contacts: true
  failure:
    affection_too_low: "contact_not_unlocked"
    npc_lost: "contact_not_unlocked"
```
