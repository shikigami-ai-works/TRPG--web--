## 作業方針

プロットをいきなり実装しない。
まず、人間にもAIにも読める仕様書へ変換する。

目標は、最終的に以下のような中間形式を作ること。

- Markdown仕様書
- YAML風の仕様DSL
- データモデル案
- MVP要件
- 画面一覧
- 状態遷移
- 実装優先順位

## フェーズ1: 未決定事項の整理

目的:

- 現時点で曖昧な点を洗い出す。
- 決めるべき順番を作る。

扱う項目:

- プレイヤーキャラ初期枠は3か4か。
- NPC連絡先の初期保存枠はいくつか。
- ロスト復活を残すか、残すならどの程度か。
- 好感度の範囲と上昇条件。
- 能力値の種類。
- ダイスルール。
- シナリオ1本の標準長さ。
- 無料シナリオと有料シナリオの差。

完了条件:

- 未決定事項リストに優先順位がついている。

## フェーズ2: 主要データの仕様化

目的:

- 実装AIがデータ構造として読める形にする。

作るもの:

- `player_character` 仕様
- `npc` 仕様
- `scenario` 仕様
- `item` 仕様
- `ending_tree` 仕様
- `billing` 仕様

例:

```yaml
player_character:
  id: "string"
  name: "string"
  status: "active | lost | memorial"
  stats:
    strength: 0
    dexterity: 0
    intelligence: 0
  inventory: []
  scenario_history: []
```

完了条件:

- 各要素について「状態」「条件」「結果」が書けている。

## フェーズ3: MVPの決定

目的:

- 最初に作る最小バージョンを決める。

MVP候補:

- プレイヤーキャラ作成
- 1本の短いシナリオ
- AIロール補助NPC 2から3人
- 好感度つきNPC 1人
- ダイス判定
- クリアとロスト
- 連絡先獲得
- NPC同伴
- シナリオ終了後の簡易エンディングツリー

この段階では、課金処理は本番決済ではなく、仕様だけ決めればよい。

完了条件:

- 「最初に実装するもの」と「後回しにするもの」が分かれている。

## フェーズ4: 画面と状態遷移

目的:

- Webアプリとして必要な画面を洗い出す。

画面候補:

- ホーム
- キャラクター一覧
- キャラクター作成
- シナリオ選択
- シナリオプレイ画面
- NPC連絡先一覧
- アイテム一覧
- エンディングツリー
- 追憶、記録画面
- ショップまたは拡張画面

完了条件:

- 各画面で表示する情報と、できる操作が決まっている。

## フェーズ5: 実装向け仕様DSL化

目的:

- プログラミングするAIが読める形へ落とし込む。

作るもの:

- `game.yaml` 相当のゲーム全体仕様
- `scenario_schema.yaml` 相当のシナリオ定義
- `character_schema.yaml` 相当のキャラクター定義
- `npc_schema.yaml` 相当のNPC定義
- `item_schema.yaml` 相当のアイテム定義

完了条件:

- この仕様を読めば、AIがWebアプリの初期実装に入れる状態になっている。

## 次回のおすすめ進行順

1. 未決定事項を並べる。
2. プレイヤーキャラクター仕様を決める。
3. NPC仕様を決める。
4. シナリオ仕様を決める。
5. ダイスとロストのルールを決める。
6. MVP範囲を決める。
7. 実装向けDSLに変換する。

## 注意点

- AIにゲームマスターを任せすぎない。
- AIは演技、反応、台詞、理由づけに使う。
- 進行、報酬、ロスト、分岐条件はゲーム側が持つ。
- 課金は強さではなく、枠、外見、物語、保存領域を中心にする。
- ロストは重く扱うが、不快な課金圧にしない。
- NPCとの関係性がこのゲームの魅力の中心になる。
