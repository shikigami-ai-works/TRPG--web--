# MVPスコープ

## MVPの目的

最初のバージョンでは、ゲームの核が面白いかを検証する。

- 探索者を作れる
- 複数の無料公開シナリオ、または利用許諾を確認した既存シナリオを読み込める
- シナリオを順次追加できる
- AI NPCと会話できる
- ダイス判定がある
- クリアまたはロストがある
- NPC好感度が変化する
- 条件を満たすとNPC連絡先を入手できる
- シナリオ終了後に結果が記録される
- 簡易エンディングツリーが更新される

## MVPに入れるもの

### プレイヤーキャラクター作成

- 名前
- 外見メモ
- 性格メモ
- 能力値
- ステータス: active / lost / memorial
- 所持アイテム
- シナリオ履歴

### 複数シナリオ読み込み

MVPではシナリオ本文の新規制作を必須にしない。
まずは無料公開シナリオ、または利用許諾を確認した既存シナリオを、ゲーム用データへ変換して複数読み込めるようにする。

この段階の目的は、シナリオ創作ではなく、以下を検証すること。

- シナリオ一覧を表示できるか
- 複数シナリオを選択できるか
- シナリオを後から順次追加できるか
- シナリオごとにNPC、シーン、判定、分岐、エンディングを分けて管理できるか
- シナリオごとのクリア履歴、ロスト履歴、エンディング到達状況を記録できるか
- 既存シナリオをゲーム用データへ変換する運用が成立するか

### シナリオ追加方式

シナリオは固定実装ではなく、データファイルとして追加する。
新しいシナリオを追加するときは、ゲーム本体のコードを大きく変更せず、シナリオ定義ファイルを追加するだけで扱える構造を目指す。

想定ディレクトリ:

```text
scenarios/
  scenario_id/
    scenario.yaml
    npcs.yaml
    scenes.yaml
    endings.yaml
    items.yaml
    license.md
```

### AIロール補助NPC

各シナリオにNPCを2〜3人配置する。
各シナリオにつき、最低1人は好感度と連絡先システムを持つNPCを用意する。

### ダイス判定

最初は「能力値 + ダイス >= 目標値」で判定する。
死に関わる判定は補正を入れて甘くする。

### NPC連絡先

条件:

- 好感度が一定以上
- NPCがロストしていない
- シナリオをクリアした

## MVPでは後回しにするもの

- 有料シナリオ販売
